# Meal Calendar Sync

Read the current weekly meal plan from `~/.claude-gombwe/data/family.json`
(under `meals` — keyed by ISO date → `{breakfast, lunch, dinner}`). Push
each meal as an event in Apple Calendar.app's **Family** calendar so it
appears on every shared device (wife's phone, kids' iPads).

**No alarms.** Calendar visibility is enough — household members glance
at their day in Calendar and see what's for dinner. Alarms feel
dictatorial, especially when trying to elicit adoption.

## Why

The gombwe dashboard shows the plan, but the household doesn't live in
the dashboard. They live in Calendar. Pushing meals into Calendar.app's
shared Family calendar means everyone in the household sees "Dinner:
butter chicken at 6:30 pm" when they look at their day. No notifications
fire. No nags.

## Source of truth

`~/.claude-gombwe/data/family.json` `meals` field. Generated by:

- The user manually via the Family tab in the dashboard
- The MCP `add_meal` tool when Claude / Discord / Telegram conversations
  set meals
- `scripts/meal-plan.mjs` (Sunday 17:00 cron — see `scripts/grocery-cron-setup.md`)

This skill is the **publish** step, not the planning step. If
`family.json` has no meals for this week, do nothing — say "no plan to
publish, run meal-plan.mjs first or add meals via the dashboard."

## What gets published

Iterate the next 7 days from today. For each day, for each slot
(breakfast, lunch, dinner):

- Skip if slot is empty for that date
- Create a Calendar.app event with:
  - **Summary**: `{Slot}: {meal name}` (e.g., `Dinner: Butter chicken`)
  - **Start time**: see "Default times" below — but use household-specific
    times if present in `family.json` under `meal_times`
  - **Duration**: 30 min (just gives the entry a visual footprint)
  - **Description**: ingredients summary
  - **No alarm** — calendar visibility only
  - **Calendar**: Family

## Default times (per-household — read from family.json if set)

If `family.json` has `meal_times: { breakfast: "07:00", lunch: "12:30",
dinner: "18:30" }` use those. Otherwise default:

| Slot | Time |
|---|---|
| Breakfast | 07:00 |
| Lunch | 12:30 |
| Dinner | 18:30 |

## Idempotency

**Critical** — this skill may run daily; never double-publish.

Before adding an event for a given date+slot, check Apple Calendar for
an existing event in Family calendar where:

- `start date` matches the slot time on that date
- `summary` starts with `Breakfast:`, `Lunch:`, or `Dinner:` (i.e., it's
  one of ours)

If found and the meal name matches what's in family.json → skip (already
there). If found but meal name differs (plan changed) → update the
existing event's summary + description. If not found → create new.

AppleScript pattern for find-or-update:

```applescript
tell application "Calendar"
  tell calendar "Family"
    set matches to (every event whose start date = (date "<slot datetime>"))
    -- iterate to find one whose summary starts with the right slot prefix
    -- if exists: update properties; else: create new
  end tell
end tell
```

## Don't

- Don't write meals into the gombwe dashboard's own calendar surface —
  they're already there (the Family tab reads family.json directly).
  This skill is one-way: family.json → Apple Calendar.
- Don't touch any calendar other than Family.
- Don't add meals from previous dates.
- Don't include the FULL recipe in the description — too long for a
  calendar entry. Just ingredients headline + cook-time hint.

## Pre-requisite (one-time, not part of this skill)

For the household to see published meals on their phones:

- Family calendar must exist in Calendar.app (default on macOS — usually
  named "Family" or "Home")
- Calendar must be shared with household members' iCloud accounts
  (Calendar.app → right-click Family → Share Calendar → invite)
- Household members must accept the invitation on their devices

If the user runs this skill and the Family calendar doesn't exist or
isn't shared, surface that as a setup gap instead of failing silently.

## Surface what happened

After running, report:

1. **N meals published** (date · slot · meal name · alarm)
2. **N meals updated** (plan changed since last publish)
3. **N meals skipped** (already current in calendar)
4. **Missing slots** — dates in the next 7 days where a slot has no
   meal in family.json. Suggest: *"Run meal-plan.mjs or add via
   dashboard to fill gaps."*

## When the user invokes this

- "Push meals to calendar"
- "Publish dinner plan"
- "Update Family calendar with meals"
- "/meal-cal"

After running, the household sees the week's meals on their phones with
prep alarms.

## Recovery

If a meal change shouldn't have published:

- Delete the offending event in Calendar.app directly, OR
- Use AppleScript to bulk-delete by date range + summary prefix:

```applescript
tell application "Calendar"
  tell calendar "Family"
    delete (every event whose start date ≥ (date "<from>") and start date ≤ (date "<to>") and summary starts with "Dinner:")
  end tell
end tell
```

The next `meal-calendar-sync` run will re-publish from the current
family.json plan.
