Agents Formation · 10 April 2026

Managed Agents and
Who They Leave Behind

By Tendai Mudavanhu · Melbourne, Australia

This week has been one of the most consequential in the short history of AI agents. In the span of four days, Anthropic blocked third-party agent harnesses, previewed a model too dangerous to release, and then launched a hosted agent platform that promises to get teams to production ten times faster. I want to talk about what managed agents are, what they mean for people building in this space, and a question that deserves more attention: who actually gets to use them?

But first, some context on where I'm coming from. At Agents Formation, a small outfit in Melbourne, we're building AI-native skill integrations for Australian businesses. Our thesis is simple: the primary interface between businesses and customers is shifting from websites and apps to callable skills that AI agents execute on behalf of users. We've been testing that thesis with Gombwe, an open-source autonomous agent runtime built on Claude. It orders groceries, switches utility plans, manages family schedules, and runs in production for a real household. It was published to npm on April 7, the day before managed agents was announced.

That timing wasn't strategic. It was coincidence. But it does mean I've been building the exact kind of system that managed agents is designed to replace.

What happened this week

April 4 Anthropic blocks OpenClaw from using Claude subscription credits. Third-party agent harnesses now require pay-as-you-go API billing. Users who were paying $20/month are suddenly facing potential costs of thousands per month for the same workflows.
April 7 Anthropic previews Claude Mythos, a model that scores 93.9% on SWE-bench and found tens of thousands of security vulnerabilities during testing. Too dangerous to release publicly. Deployed to 11 major companies for code security through Project Glasswing.
April 7 Gombwe v0.1.0 published to npm. An autonomous agent runtime with 15 skills, 21 native tools, multi-channel support, and a live grocery ordering flow. Built on Claude.
April 8-9 Anthropic launches managed agents in public beta. Cloud-hosted agent infrastructure with sandboxed environments, credential vaults, and MCP server integration. Notion, Rakuten, and Asana among early adopters.

Read sequentially, a pattern emerges. Anthropic shut down third-party harnesses, demonstrated model superiority, and then offered their own hosted alternative. This suggests a platform play.

What managed agents actually are

Anthropic published an engineering blog post describing the architecture behind managed agents. The key insight is a three-way decoupling:

  1. Session (state): a durable, append-only event log that persists across crashes and can be queried by the agent
  2. Harness (brain): a stateless inference loop that calls Claude and routes tool calls, rebootable without losing context
  3. Sandbox (hands): execution environments provisioned on-demand, exposed as tools via a generic execute(name, input) → string interface

It's genuinely good engineering. The decoupling means any component can fail and be replaced without losing the session. The sandbox interface is intentionally generic. In Anthropic's words: "We're opinionated about the shape of these interfaces, not about what runs behind them." MCP servers, shell scripts, browser automation: it all looks the same to the harness.

From a developer's perspective, you define your agent's tasks, connect your tools via MCP servers, configure an environment, and Anthropic handles the rest: no provisioning containers, no managing state, no building retry logic. For someone who has never deployed an agent before, this is transformative.

What's missing

Several developers have noted that managed agents can't run on a schedule. There's no cron. No heartbeat. You can't say "wake up every 30 minutes and check for new tasks." The agent runs when you trigger it via API or chat, and then it stops. For anyone building always-on assistants (the kind that monitor, react, and act autonomously) this is a significant gap.

Gombwe has had cron scheduling since day one. It's how our grocery ordering works: the agent wakes up on schedule, checks the meal plan, extracts ingredients, compares prices, and proposes an order. The user confirms via Discord or Telegram. No manual trigger.

Anthropic has teased upcoming features: outcomes (self-evaluation), multi-agent orchestration, and persistent memory across sessions. These will address some of the gaps. But they're not available yet.

The access question

Here's what I keep coming back to: who gets to use these agents?

Managed agents require a Claude account. The tools, MCP servers, and sandboxed environments are only accessible to paying subscribers or API customers. A free Claude user cannot use any of this. The minimum entry point is a Pro subscription at $20 per month, or an API key with credit.

For a developer in San Francisco or a product team at Notion, this is trivial. For a family in Harare who wants to compare grocery prices across OK Zimbabwe and Bon Marché, it's a month's mobile data budget. For the 600 million people projected to be online in Africa by 2030, many of whom may interact with commerce through AI agents before they ever use traditional e-commerce, it's a wall.

The $20/month subscription isn't a price. It's a filter. And it filters out most of the world.

This matters because the tools are built by the community. Anthropic doesn't build Australian grocery search tools, or Zimbabwean ones, or Nigerian ones. They build the infrastructure: the sandbox, the harness, the session persistence. The domain-specific tools come from developers like me, publishing MCP servers, writing for free, receiving no revenue share. Anthropic lists them in their directory. The customer pays Anthropic. The tool provider gets nothing.

That's the deal. And it's not necessarily unfair: Anthropic provides the model, the infrastructure, the distribution. But it does mean the tools that community developers build are only accessible to people who can afford a subscription. Sandbox compute is included for subscribers, and billed at $0.08/hr for API users. What's less clear is what happens when an enterprise builds a managed agent product and advertises it to the public: who pays for those end user sessions? The enterprise? The end user? That billing model hasn't been addressed yet.

What we're building instead

I'm not arguing against managed agents. For enterprise teams, they're excellent. The architecture is sound. The developer experience is a genuine step forward. Anthropic is solving a real problem for a real audience.

But I think there's a parallel path that needs to exist. One where the tools are accessible to anyone, not just paying subscribers. Where the agent lives where the customer already is: WhatsApp, Telegram, Discord, SMS, not inside a chat window that requires a subscription. Where data stays in the country it comes from. Where the pricing can be per-transaction, or free, or whatever the market demands.

That's what we're building at Agents Formation. We call it Gombwe Managed Agents: a multi-tenant platform where customers interact with AI agents over messaging channels to search, compare, and buy from real businesses. Starting with Australian grocery (Woolworths and Coles), expanding to flights, energy, insurance, and eventually African markets.

The core differences:

  • No subscription required. Free tier for search and comparison. Pay per transaction if you want to buy.
  • Any channel. WhatsApp, Telegram, Discord, web. Not locked to Claude's interface.
  • Any model. Claude for complex reasoning, Haiku for simple lookups, open-source models where cost matters. Match the model to the task.
  • Australian data sovereignty. Hosted on AWS Sydney. Customer data stays in Australia.
  • End-to-end transactions. Not just search and compare, but actually add to cart, confirm, and buy. The transactional layer that no MCP standard addresses yet.
  • Direct retailer relationships. We're in Melbourne. We can negotiate API access, affiliate commissions, and partnerships that a company in San Francisco never will.

The bigger picture

I think we're at an inflection point. The runtime layer (the brain, the sandbox, the session management) is getting commoditised. Anthropic offers it. Google will. OpenAI will. That's infrastructure.

The value is in two places: the domain-specific tools (the MCP servers that know how to search Woolworths, query an energy retailer, or book a flight) and the access layer (who can use these tools, through what channels, at what price).

Anthropic is building excellent infrastructure for the first world. Someone needs to build the access layer for everyone else. We think that someone should be close to the markets they serve, building tools that reflect local knowledge, accessible at local price points, over local channels.

That's the bet. We could be wrong about the timing. But the direction seems clear.

Gombwe is open source and in production. The Australian MCP Registry lists our published servers. We're looking for early users, retail partners, and anyone thinking about agent commerce in Australia or Africa.
