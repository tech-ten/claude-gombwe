Agents Formation · 10 April 2026

Managed Agents and
Who They Leave Behind

By Tendai Mudavanhu · Melbourne, Australia

This has been one of the most consequential weeks in the short history of AI agents. Anthropic made a series of moves that clarify where they see the future of agent infrastructure. For those of us building in this space, it opens up important questions about where agents should run, who should have access to them, and what role the model plays versus the infrastructure around it.

Some context on where I'm coming from. At Agents Formation, a small outfit in Melbourne, we're building AI-native skill integrations for Australian businesses. We've been testing that thesis with Gombwe, an open-source autonomous agent runtime built on Claude. It orders groceries, compares prices across Woolworths and Coles, manages family schedules, and runs in production for a real household. It was published to npm on April 7, the day before managed agents was announced. That timing was coincidence, but it means I've been building the exact kind of system that managed agents is designed to replace.

What happened this week

April 4 Anthropic updates billing for third-party harnesses. Tools like OpenClaw now require pay-as-you-go API billing instead of subscription credits. This shifts the economics for developers who were building on top of Claude subscriptions.
April 7 Anthropic previews Claude Mythos, a model that scores 93.9% on SWE-bench and found tens of thousands of security vulnerabilities during testing. Too dangerous to release publicly. Deployed to 11 major companies for code security through Project Glasswing.
April 7 Gombwe v0.1.0 published to npm. An autonomous agent runtime with 15 skills, 21 native tools, multi-channel support, and a live grocery ordering flow. Built on Claude.
April 7 NVIDIA's NemoClaw launches as an open-source agent framework that runs locally on Docker Desktop. A different approach to the same problem: agents that run on your machine, not in someone else's cloud.
April 8-9 Anthropic launches managed agents in public beta. Cloud-hosted agent infrastructure with sandboxed environments, credential vaults, and MCP server integration. Notion, Rakuten, and Asana listed as early adopters.

It's a busy week that signals where the industry is heading. Anthropic is investing heavily in hosted agent infrastructure. At the same time, NVIDIA, OpenClaw, and Gombwe are all building agent runtimes that run locally. Two different visions for how agents should work, and both have merit.

What managed agents actually are

Anthropic published an engineering blog post describing the architecture behind managed agents. The key insight is a three-way decoupling:

  1. Session (state): a durable, append-only event log that persists across crashes and can be queried by the agent
  2. Harness (brain): a stateless inference loop that calls Claude and routes tool calls, rebootable without losing context
  3. Sandbox (hands): execution environments provisioned on-demand, exposed as tools via a generic execute(name, input) → string interface

It's genuinely good engineering. The decoupling means any component can fail and be replaced without losing the session. The sandbox interface is intentionally generic. In Anthropic's words: "We're opinionated about the shape of these interfaces, not about what runs behind them." MCP servers, shell scripts, browser automation: it all looks the same to the harness.

For someone who has never deployed an agent before, the value proposition is real: define your tasks, connect your tools, and Anthropic handles containers, state, and retry logic. That is a genuine step forward in developer experience.

The value question

Managed agents bundle the model with infrastructure. It's worth understanding what each part of that bundle provides:

An agent has three parts: the model (intelligence), the tools (your code), and the runtime (infrastructure to run your code). Anthropic's unique contribution is the model. The sandbox, harness, and session store are standard infrastructure components: containers, loops, and databases. Developers can run these on their own machines, a VPS, or their existing cloud provider.

Managed agents bundle all three together as a convenience. Sandbox compute is included for Claude subscribers ($20/month minimum) and billed at $0.08/hr for API users. The convenience is real, especially for teams that want to skip the infrastructure setup. But it's worth noting that the infrastructure components are widely available. Enterprises already run containers on their cloud providers. Individual developers can run them on their laptops.

The community is already exploring this alternative. OpenClaw built an open-source agent harness. NVIDIA released NemoClaw, which runs on Docker Desktop. Gombwe has been running autonomously on a home machine since before managed agents was announced. These projects demonstrate that the harness layer is something the community can and will build independently.

The efficiency question

There is an interesting architectural trade-off worth considering. Managed agents route every tool call through the model: every data fetch and every decision goes through Claude. This is powerful for complex reasoning chains, but much of what agents do is deterministic work that does not require intelligence.

Consider grocery shopping. Searching Woolworths for "chicken breast" is an HTTP request. Comparing prices across two stores is a sort. Adding items to a cart is a form submission. None of this needs a large language model. A script does it in milliseconds, for free.

What does need intelligence is the ambiguous, open-ended work: "plan me a week of meals under $100" or "we're running low on pantry staples, figure out what to order." That requires reasoning about preferences, budgets, and trade-offs. That is where the model earns its tokens.

Gombwe takes a different approach by separating these concerns. Scripts handle deterministic work (search, compare, add to cart). The results are injected into the prompt as context. Claude reasons about the results and talks to the user. The model only runs when intelligence is actually needed. This hybrid approach can be more efficient for use cases where most of the work is structured and predictable.

The most efficient agent uses the model for reasoning and scripts for everything else.

The local access question

Managed agents run in Anthropic's cloud, in a sandboxed container with no access to the user's machine. This is excellent for security and isolation, but it does create constraints for certain use cases.

Gombwe's grocery flow depends on local access: a logged-in browser session with saved store credentials, personal data files on disk, scripts that interact with local services. A remote sandbox cannot reach any of this. To use managed agents for the same workflow, you would need to upload credentials, personal data, and browser sessions to someone else's infrastructure, hosted in the US.

For consumer use cases that involve personal accounts, local files, or saved sessions, the agent needs to be where the data is. A cloud sandbox works well for cloud-native tasks like code generation or data analysis. But for personal agents that interact with the real world on behalf of a specific user, local execution offers advantages that a remote sandbox cannot match.

The access question

Beyond the architectural considerations, there is also a question of accessibility.

Managed agents require a Claude subscription ($20/month minimum) or API credits. A free Claude user cannot use any of this. The tools, MCP servers, and sandboxed environments are behind a paywall.

For a developer in San Francisco, this is trivial. For a family in Harare who wants to compare grocery prices across OK Zimbabwe and Bon Marché, it's a month's mobile data budget. For the 600 million people projected to be online in Africa by 2030, many of whom may interact with commerce through AI agents before they ever use traditional e-commerce, it is a wall.

This is particularly relevant because the domain-specific tools are built by the community. Anthropic builds excellent infrastructure: the sandbox, the harness, the session persistence. But the tools that make agents useful for specific markets come from independent developers, publishing MCP servers and building integrations for their local contexts. There's an opportunity for better alignment between the value these developers create and how they're compensated.

Sandbox compute is included for subscribers and billed at $0.08/hr for API users. What's less clear is what happens when an enterprise builds a managed agent product and exposes it to the public: who pays for those end user sessions? The enterprise? The end user? That billing model hasn't been addressed yet.

Where managed agents do make sense

I don't want to be dismissive. There are legitimate use cases where a cloud sandbox is the right choice:

  • Safety-critical workloads. Models like Mythos that find security vulnerabilities need controlled, isolated environments. Running those on a personal machine is a risk.
  • Long-running background tasks. A coding agent that works for hours needs compute that doesn't sleep when a laptop lid closes.
  • Non-technical teams who need agents but cannot install Node.js, configure Docker, or manage infrastructure.
  • Rapid prototyping. If you want to test an idea in an afternoon without setting up infrastructure, managed agents remove friction.

These are real needs. Alongside these, there's room for open runtimes on infrastructure developers already have: their own machines, their cloud providers, their Docker Desktop. Both approaches can coexist.

What we're building alongside

We think there's a complementary path worth exploring. One where the agent runs where the data is. Where deterministic work is handled by scripts, and the model is called only when reasoning is needed. Where the tools are accessible to anyone, including people who can't afford a subscription. Where the agent lives where the customer already is: WhatsApp, Telegram, Discord, SMS.

That's what we're building at Agents Formation. We call it Gombwe Managed Agents: a multi-tenant platform where customers interact with AI agents over messaging channels to search, compare, and buy from real businesses. Starting with Australian grocery (Woolworths and Coles), expanding to flights, energy, insurance, and eventually African markets.

The core differences:

  • No subscription required. Free tier for search and comparison. Pay per transaction if you want to buy.
  • Any channel. WhatsApp, Telegram, Discord, web. Not locked to Claude's interface.
  • Any model. Claude for complex reasoning, Haiku for simple lookups, open-source models where cost matters. Match the model to the task.
  • Scripts for deterministic work. The model only runs when intelligence is needed. Everything else is code.
  • Local-first. Runs on your machine, accesses your data, uses your browser sessions. No uploading credentials to a remote sandbox.
  • Australian data sovereignty. Hosted on AWS Sydney. Customer data stays in Australia.
  • End-to-end transactions. Not just search and compare, but actually add to cart, confirm, and buy.

The bigger picture

Anthropic's managed agents represent a significant step forward for the industry. The architecture is sound, the engineering is impressive, and for many teams it will be the right choice. At the same time, multiple projects are demonstrating that agent runtimes can run locally, openly, on hardware developers already own. The ecosystem is big enough for both approaches.

We believe the most exciting opportunities are in two places: the domain-specific tools (the code that knows how to search Woolworths, query an energy retailer, or book a flight) and the access layer (who can use these tools, through what channels, at what price). These can be built on top of any infrastructure, managed or self-hosted.

We're excited about a future where agents run where the data is, use the model only when reasoning is needed, and are accessible to anyone regardless of what they can afford. Managed agents and open runtimes will both play a role in getting there. We could be wrong about the timing. But the direction seems clear.
