Managed Agents and
the Case for Personal Agents

By Tendai Mudavanhu · Melbourne, Australia

Two days ago, Anthropic launched managed agents. It's a significant moment for anyone building in the agent space, and I've been thinking about what it means for projects like Gombwe, our open-source personal agent that orders groceries, manages family schedules, and runs on a home machine in Melbourne. Managed agents raises the bar for enterprise agent infrastructure. It also clarifies where independent projects like ours fit in.

The week in context

When Anthropic moved OpenClaw to API-only billing on April 4, the direction was clear. A managed agent offering was coming. I expected it would take weeks, maybe months. It took four days.

I published Gombwe to npm on April 7, the day before the announcement. Coincidence, not strategy. But the timing made the impact personal. I had spent months building an autonomous agent runtime: a completion loop with retries and verification, multi-channel support via Discord and Telegram, session management, 15 declarative skills, cron scheduling, and a live grocery ordering flow. Then Anthropic published their engineering blog post describing a three-way decoupled architecture with crash-resilient sessions, stateless harnesses, context engineering that reduced P95 latency by over 90%, and model-harness co-optimisation that improves structured task performance by 10 points.

It is excellent engineering. For enterprise deployments, it sets a new standard. For personal agent projects like Gombwe, it sharpens the question of where we add unique value.

What Anthropic got right

Let me be specific about what managed agents delivers, because it's more than "hosted containers":

  • The harness is not just a loop. Anthropic's research documents specific failure modes that destroy naive agent implementations: over-ambition, premature completion, context rot as token count grows. Their harness handles compaction, just-in-time retrieval, structured note-taking, and sub-agent summarisation. I built a completion loop with retry logic. They built a hardened orchestration layer informed by research I hadn't considered.
  • Security is a real advantage. OpenClaw had 138 CVEs in 2026, including a 9.9/10 severity vulnerability. 824 malicious skills were planted on their marketplace. Managed agents' sandboxed execution with credential vaults is genuinely safer than running agents locally with --dangerously-skip-permissions, which is exactly what Gombwe does.
  • Time-to-production matters. Rakuten deployed agents across five departments in a week each. Sentry shipped a bug-fix agent with a single engineer. 95% of internally developed AI pilots fail to reach production. Managed agents removes the infrastructure barrier that kills most agent projects.
  • MCP ties it together. Anthropic donated the Model Context Protocol to the Linux Foundation, making tool integration an open standard. Managed agents consumes MCP servers natively. This is generous and strategic in equal measure: it means every tool the community builds feeds their ecosystem, but it also means those tools are portable to any platform that speaks MCP.

An honest comparison

After researching extensively, here is where I landed on the strengths and limitations of each approach:

Where managed agents wins

  • Reliability. 73% of enterprise agent deployments experience failures in their first year. Agents show 91% failure rates on complex tasks. Anthropic's harness is engineered specifically to handle these failure modes.
  • Security. Sandboxed execution with credential vaults, SOC 2 Type II compliance, and isolation by design. Local agents running with full system access are a proven attack surface.
  • Time-to-production. Define tasks, connect tools, deploy. No infrastructure to manage, no containers to provision, no retry logic to build.
  • Context engineering. Prompt caching, compaction, just-in-time retrieval, and model-harness co-optimisation that third parties cannot replicate because Anthropic controls both sides.
  • Multi-agent orchestration. Anthropic's research shows 90% improvement over single-agent performance. Their coordination, error propagation handling, and token optimisation represent deep engineering.
  • Background execution. Agents that run for hours on tasks that would outlast a laptop battery or survive a lid close.

Where local/personal agents win

  • Local data access. Browser sessions, saved credentials, personal files, household data. A remote sandbox cannot reach these without uploading them to third-party infrastructure.
  • Deterministic efficiency. Scripts handle structured work (search, compare, sort) in milliseconds for free. No tokens burned on work that doesn't need reasoning.
  • Zero marginal cost. Runs on existing hardware using an existing subscription. No per-session billing, no sandbox fees.
  • Scheduling. Cron-based wake-up, proactive monitoring, and autonomous action without manual triggers. Managed agents runs when called.
  • Channel integration. Lives in Discord, Telegram, WhatsApp, where users already are. End users never need to know about AI subscriptions or Claude.
  • Data sovereignty. Personal data stays on the user's machine or in their chosen jurisdiction. No dependency on US-hosted infrastructure.
  • Accessibility. Available to users who cannot afford $0.08/hr sandbox fees or $20/month subscriptions, including emerging markets.

Where personal agents lead

After sitting with this for two days, I think the honest framing is not "local vs cloud" or "open vs managed." The real distinction is between enterprise agents and personal agents. Managed agents is exceptional at the first. It's not designed for the second.

A personal agent is something different from a deployment. It's an always-on assistant that lives in your household's Discord server, knows your family's dietary preferences, wakes up on a schedule to check the meal plan, and orders groceries from your logged-in Woolworths account using your saved payment method. Nobody in my family knows or cares that Claude exists. They text "we need milk" and it happens.

Managed agents can't do this today. Not because of a technical limitation, but because the product is aimed elsewhere. Specifically:

  • Channels. Managed agents lives inside Claude's interface or behind an API. Gombwe lives in Discord and Telegram, where my family already is. Building a thin channel adapter on top of the managed agents API is possible, but it's not what the product offers out of the box.
  • Scheduling. Managed agents runs when triggered. Gombwe wakes up on cron, checks the meal plan, extracts ingredients, compares prices, and proposes an order. No manual trigger.
  • Private MCP servers. Some tools I don't want to share with any cloud platform. My family's meal plans, pantry state, dietary restrictions, and purchase history live in a local MCP server that Claude accesses but Anthropic never sees. Managed agents' credential vault is secure, but there are workloads where keeping data entirely local is the right choice.
  • Deterministic efficiency. Searching Woolworths for "chicken breast" is an HTTP request. Comparing prices is a sort. Adding to cart is a form submission. Gombwe runs these as scripts and injects the results into the prompt. The model only runs when reasoning is actually needed. Managed agents routes every action through Claude, which is powerful for complex chains but burns tokens on work that doesn't require intelligence.
  • Zero cost for the end user. Gombwe runs on my machine using my existing Claude subscription via claude -p. No API costs, no per-session billing, no $0.08/hr sandbox fees. For a household that sends 20 messages a day to their grocery agent, managed agents would cost real money. Gombwe costs nothing beyond what I already pay.

The case for personal agents

I think managed agents makes the term "AI agent" an enterprise product category. What Gombwe does is different enough that it probably needs a different name. A personal agent, or maybe a personal assistant that happens to use AI when it needs to. The AI is an implementation detail, not the product.

A personal agent:

  • Runs where the data is, on the user's own machine
  • Connects to channels the user already uses (Discord, Telegram, WhatsApp)
  • Handles deterministic work with scripts, calls the model only for reasoning
  • Keeps private data in private MCP servers the user controls
  • Runs on a schedule without manual triggers
  • Costs nothing beyond an existing subscription, or runs on open-source models for zero cost

This is a small market compared to enterprise. But it's a real one. Not everyone needs Anthropic's infrastructure. Some people need an agent that knows their family, runs on their laptop, and texts them on Discord when the chicken is on sale.

What this means for Agents Formation

We're a small outfit in Melbourne building Gombwe and a set of MCP servers for Australian services. Our focus is clear: a personal agent platform and an independent comparison layer for Australian commerce.

That includes:

  • Household automation. Grocery ordering, meal planning, family scheduling through messaging channels.
  • Local-first workflows. Agents that need access to logged-in browser sessions, local files, or hardware that a remote sandbox can't reach.
  • Cost-sensitive deployments. Use cases where $0.08/hr per session is a real barrier, including emerging markets where a subscription costs more than a month's mobile data.
  • Private data workloads. MCP servers that expose personal information the user doesn't want leaving their machine.
  • Australian data sovereignty. Customer data that needs to stay in Australia, hosted on AWS Sydney rather than US infrastructure.

We're also rethinking what Gombwe is architecturally. Today it wraps Claude Code as a harness. In the future, it could be a channel and orchestration layer that works with managed agents for complex reasoning and local scripts for deterministic work. The best architecture might not be "local vs cloud" but "local and cloud, each doing what it does best."

The layer nobody is building

There's a bigger argument here that goes beyond personal agents versus managed agents. Our thesis at Agents Formation has always been that the primary interface between businesses and customers is shifting from websites and apps to callable skills that AI agents execute on behalf of users. Managed agents accelerates that shift. But it also reveals a gap that neither Anthropic nor the retailers will fill.

Consider: could Woolworths build an MCP server and list it in Claude's directory? Yes. Would customers then be able to say "order my groceries" inside Claude? Yes. But Woolworths will never build a tool that lets Claude tell the customer "Coles has chicken cheaper this week." No retailer will build the comparison layer that works in the customer's interest. They'll build endpoints that showcase their own products, their own promotions, their own upsells.

Anthropic won't build it either. They build infrastructure. They don't build Australian grocery comparison tools.

Someone needs to build the independent layer that sits between customers and businesses: the comparison, the aggregation, the recommendation that serves the customer, not the retailer. That's what the Australian MCP Registry is becoming. Not a catalogue of our own servers, but a marketplace where Australian businesses either show up with their own endpoints or get represented by tools that compare them anyway.

Managed agents makes this more urgent, not less. As agents become the default interface for commerce, the businesses that have MCP endpoints will be discoverable. The ones that don't will be invisible to a generation of customers who never visit a website. The question is whether retailers build those endpoints themselves, or whether independent platforms build them first.

The bigger picture

Anthropic built something impressive this week. Managed agents, combined with their donation of MCP to the Linux Foundation, creates a platform that will accelerate enterprise agent adoption significantly. The engineering is deep, the security model is sound, and the time-to-production advantage is real. For most teams building agent-powered products, it is the right choice.

Personal agents occupy a different space: the always-on household assistant, the local-first workflows, the zero-cost use cases that matter to individuals and families. Beyond that, there's an opportunity that applies regardless of infrastructure: the independent comparison layer that works for customers, not retailers. This can be built as a managed agent, as a local tool, or as an MCP server inside Claude's ecosystem. No retailer has an incentive to build it, and it hasn't been built yet.

That's where we're focused. The tools, the channels, the registry, the personal experiences, and the comparison layer. We think the people building these should be close to the markets they serve, with local knowledge, accessible pricing, and channels that reach everyone.
